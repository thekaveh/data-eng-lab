# Documentation Overhaul — Design Spec

**Topic:** Overhaul of README.md, GitHub Wiki, and .io MkDocs site into a unified system with single source of truth.

**Date:** 2026-07-06

**Approach:** MkDocs-driven — `docs/` source tree feeds `.io` site, README, and wiki automatically.

---

## 1. Architecture Overview

**Single source of truth:** All content lives in `docs/`. README.md and GitHub Wiki are *derived products* generated on every CI deploy — never edited manually.

### Flow Diagram

```
                  Source of Truth (single copy, edited by humans)
           ┌───────────────────────────────────────────────────┐
           │   docs/                                           │
           │     ├── index.md                → README.md       │
           │     ├── getting-started.md                       │
           │     ├── lakehouse.md                           │
           │     ├── datasets.md                            │
           │     └── scenarios/*.md                         │
           │     └── spark-apps/*.md                        │
           └──────────────┬────────────────────────────────┘
                          │
           mkdocs build  │  parse & extract
               ┌────────┘  ┌──────────┐      ┌────────────────┐
               v            │ scripts  │      │ scripts:       │
        .io Site (GitHub  │ render-  │      │ render-readme/ │
        Pages)            │ wiki.py  │      │ render-readme  │
                           └────┬─────┘      └────────────────┘
                                │
                        wiki-clone/
                           → wiki repo push
```

**What each surface becomes:**

| Surface | Role | Who edits directly? |
|---------|------|---------------------|
| `docs/*.md` | Source of truth. MkDocs renders into `.io` site with dark theme, nav, cross-links, diagrams. | **You — always.** This is the only place you edit. |
| GitHub Wiki (wiki repo) | Mirror of all non-index content (scenario docs, lakehouse, spark-app docs). Generated as raw Markdown pushed by the build script on every deploy to `main`. | **Never** — editing here would be overwritten on the next CI run. |
| Top-level `README.md` | Auto-derived from `docs/index.md`. Shows same overview + quick-start as the .io home page. | **Never** — auto-generated by `render-readme.py` on every CI run. |

### Per-Document Template

Every scenario and spark-app doc follows this consistent outline:

1. **Overview** — What it does, why it exists in the lakehouse.
2. **Data Model** — Input schema (source) / Output schema (Iceberg tables, paths, layers).
3. **Architecture** — Embedded diagram + explanation of data flow, services, dependencies.
4. **Notebooks** — Walkthrough of Zeppelin Scala + Jupyter PySpark notebooks (identical logic, different languages).
5. **Orchestration** — Airflow DAG (ID, schedule, tasks) or long-running streaming note.
6. **Usage** — Step-by-step run instructions, prerequisites, verification commands.
7. **See Also** — Cross-links to upstream/downstream scenarios and related datasets.

Each doc has a **See Also** section at the bottom with bidirectional cross-links (if A references B as upstream, B lists A as downstream consumer).

## 3. README + Wiki Sync Design

### README

`scripts/render-readme.py` extracts content from `docs/index.md` and writes it to the root `README.md` on every CI deploy:

1. Extract the overview paragraph (first section after title).
2. Extract the "By the Numbers" table.
3. Extract the "Quick Start" code block.
4. Extract the "Explore" cards with navigation links to each sub-section.
5. Extract the scenario catalog table with all 19 entries.

**Manual edits to README.md are not allowed** — they would be overwritten on the next CI run. The file is checked into git but any human change would immediately be replaced. The MkDocs nav structure defines what appears in the rendered README.

### GitHub Wiki

`scripts/render-wiki.py` mirrors all `docs/` content (except `index.md`, which is README) into raw Markdown files pushed to `data-eng-lab.wiki.git`:

1. Copy `docs/lakehouse.md` → wiki `lakehouse.md`.
2. Copy `docs/getting-started.md` → wiki `getting-started.md`.
3. Copy all `docs/scenarios/*.md` → wiki `scenario-<name>.md` (flattened).
4. Generate `_Sidebar.md` from MkDocs nav (same hierarchy, same links).

**Manual edits to the wiki are not allowed** — any change would be overwritten on the next deploy. The wiki repo only receives content from this script.

### CI Workflow

```yaml
on: { push: { branches: [main] } }

jobs:
  build-io-site:
    steps:
      - checkout, setup python, pip install mkdocs-material
      - mkdocs build
      - deploy to GitHub Pages

  sync-readme-and-wiki:
    needs: [build-io-site]
    steps:
      - checkout
      - python scripts/render-readme.py   → root README.md (tracked, auto-committed)
      - checkout data-eng-lab.wiki.git   → wiki-clone/
      - python scripts/render-wiki.py    → wiki-clone/
      - cd wiki-clone && git add + commit + push origin master
```

The two jobs run in parallel (wiki sync only *needs* the build to succeed; both run on every push to `main`). If wiki sync fails (network hiccup pushing to wiki repo), the .io site still deploys — only a warning is raised. README updates succeed as part of the main repo workflow.

## 4. Diagram Generation Approach

### Architecture Diagrams

All diagrams are generated via the `architecture-diagram` skill — dark theme by default, with light-mode toggle available (matching MkDocs Material palette toggle).

| Document | Diagram Scope | Source | Format | Location |
|----------|--------------|--------|--------|----------|
| `docs/index.md` (home page + README source) | Full-stack: Spark, Zeppelin, JupyterHub, Airflow, Jenkins, MinIO, Iceberg REST, Trino, Redpanda, medallion tiers | Atlas platform configs, lakehouse configs | HTML/SVG | `docs/architectures/architecture.html` |
| `docs/lakehouse.md` | Lakehouse flow: landing → bronze → silver → gold, MinIO buckets, Iceberg REST + PostgreSQL | lakehouse configs, infra/service definitions | HTML/SVG | `docs/architectures/lakehouse.html` |
| Each `docs/scenarios/*.md` | Per-scenario data flow: source → processing → sink (batch, streaming, CDC, etc.) | Each scenario notebook, dag.py files | HTML/SVG | `docs/architectures/scenarios/<name>.html` |
| Each `docs/spark-apps/*.md` | CI/CD flow: Jenkins → shade → JAR → Airflow → Spark cluster → Iceberg sink | pom.xml, Jenkinsfile, dag.py | HTML/SVG | `docs/architectures/apps/<name>.html` |

### Generation Mechanics

`scripts/generate-diagrams.py` parses YAML frontmatter from each doc for source paths, data flow descriptions, and output table names, then renders an HTML/SVG diagram using the `architecture-diagram` skill. The script embeds the resulting reference path back into the doc automatically.

All diagrams use the indigo primary (`#4338ca`) and amber accent (`#f59e0b`) color palette matching `docs/css/custom.css`.

### Cross-Linking

- Every diagram file lives alongside its parent scenario/doc in `architectures/`.
- Embedding uses relative paths (`architectures/<name>.html`), so moving docs still keeps diagrams linked correctly (handled by the generation script via frontmatter metadata).
- "See Also" sections in each doc use relative markdown links (`../scenarios/sibling.md`) and are verified at build time.
